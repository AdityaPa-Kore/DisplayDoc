Purpose:

This script validates and processes Tax ID input from users. It handles both numeric and spoken word inputs (e.g., "one two three" → "123"), ensuring the Tax ID is exactly 9 digits before proceeding.

What Happens:

  1. Input Conversion: The convertToDigits() function takes the user's Tax ID input and converts it to a standardized numeric format:
    • Converts spoken numbers to digits ("one" → "1", "two" → "2", etc.)
    • Handles common speech recognition errors ("to"/"too" → "2")
    • Removes all non-numeric characters (spaces, dashes, etc.)
    • Validates the result is exactly 9 digits
  2. Retry Counter Management: Increments Taxcounter to track how many times the user has attempted to provide a Tax ID. This prevents infinite retry loops.
  3. Validation Logic:
    • If Invalid: Sets TaxID flag to "false", deletes the entity to prompt again
      • If this is the 3rd attempt (Taxcounter > 2): Routes to agent transfer
      • Otherwise: Routes back to Tax ID collection for retry
    • If Valid: Sets TaxID flag to "true" and routes to Payment ID collection
  4. Session Tagging: When max retries are exceeded, adds a session tag PaymentDetailsCallFlow_TaxId_invalid for analytics and reporting.

Why This Matters:

Tax IDs are critical identifiers in payment lookups. The conversion logic is essential because voice channels often capture spoken numbers as words. The retry limit (3 attempts) balances user experience with preventing abuse, while the validation ensures downstream APIs receive properly formatted 9-digit Tax IDs.

function convertToDigits(input) {

    if (typeof input !== "string") {

        input = String(input); // Convert to string if it's not already

    }



    const wordToDigit = {

        'zero': '0', 'one': '1', 'two': '2', 'three': '3', 'four': '4',

        'five': '5', 'six': '6', 'seven': '7', 'eight': '8', 'nine': '9',

        'to': '2', 'too': '2'

    };



    // Replace word representations of numbers with their digit equivalents

    input = input.toLowerCase().replace(/(zero|one|two|three|four|five|six|seven|eight|nine|to|too)/g, match => wordToDigit[match]);



    // Remove non-digit characters

    let result = input.replace(/[^0-9]/g, '');



    return (result.length === 9) ? result : 'Invalid Input';

}

This script handles Claim ID validation as an alternative lookup path. Users can provide Claim ID instead of Tax ID + Payment ID combination.

What Happens:

  1. Input Processing: The convertToDigitIE() function processes Claim ID with flexible validation:
    • Converts spoken numbers to digits
    • Removes periods (common in claim numbers)
    • Accepts variable length (6-21 characters) unlike fixed 9-digit Tax ID
  2. Flexible Validation: Claim IDs have more lenient validation because they vary by payer and claim type. The length check (6-21 chars) accommodates different claim numbering systems.
  3. Routing Decision:

Why This Matters:

Claim ID lookup provides an alternative when users don't have Tax ID or Payment ID readily available. This improves user experience by offering multiple ways to retrieve payment information. The UseClaimID flag ensures the correct API endpoint is called downstream.

Purpose:

This script handles API failure scenarios by implementing retry logic. It attempts to recover from transient failures without immediately escalating to an agent.

What Happens:

  1. Retry Counter Check: Examines RetryPaymentAPI counter to determine if retry is allowed (max 2 attempts).
  2. Error Analysis: Checks the API response status code:
    • 401 (Unauthorized): Token expired, routes to token refresh node
    • 500 (Server Error): Transient failure, eligible for retry
    • 404 (Not Found): Invalid Tax ID/Payment ID, no retry needed
    • Timeout: Network issue, eligible for retry
  3. Retry Decision:
    • If retry count < 2: Increment counter and route back to API call
    • If retry count >= 2: Route to agent transfer with error context
  4. Context Preservation: Maintains all collected entities (Tax ID, Payment ID) so retry doesn't require re-collection.

Why This Matters:

APIs can fail temporarily due to network issues, service restarts, or high load. Automatic retry improves success rates without bothering the user. However, limiting to 2 retries prevents long waits. The 401 token refresh handling is critical - expired tokens are expected in long conversations, and auto-refresh provides seamless user experience.
